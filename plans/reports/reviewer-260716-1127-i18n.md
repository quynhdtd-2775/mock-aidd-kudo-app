# Review: Multilingual (VN/EN) i18n Feature

Scope: lib/i18n/*, i18n/request.ts, next.config.ts, app/layout.tsx, messages/*.json + message-keys.test.ts,
supabase/migrations/20260716090000_profile_language.sql + seed.sql additions, components/language-dropdown/*,
components/home/{site-header,site-footer,user-menu,language-switcher,hero-*,awards-section,award-card,sunkudos-section}.tsx,
app/home-page-saa/page.tsx.

## Score: 8/10 — 0 critical, 1 major, 3 minor

## Critical
None.

## Major

**1. Locale cookie has no `maxAge`/`expires` — guest preference doesn't survive the session**
`lib/i18n/set-locale-action.ts:18` and `load-preferred-locale.ts:33` both call
`cookieStore.set(LOCALE_COOKIE, locale, { path: "/", sameSite: "lax" })` with no `maxAge`. Without it the
cookie is session-scoped and disappears on browser close. Design intent (plan.md, clarifications.md) is
"guests use the cookie only" as their persistence mechanism — with no maxAge that mechanism silently reverts
to `DEFAULT_LOCALE` every new browser session, which reads as a broken feature to a guest who picked "EN"
yesterday. Not flagged in clarifications.md (an open gap, not an explicit decision). Fix: add
`maxAge: 60 * 60 * 24 * 365` (or similar) to both `cookieStore.set` calls — cheap, one-line fix in two files.

## Minor

**2. `Locale` type duplicated — two independent sources of truth**
`lib/i18n/locale-config.ts` derives `Locale` from `SUPPORTED_LOCALES` (`as const` tuple). Separately,
`components/language-dropdown/language-dropdown-data.ts:1` declares `export type Locale = "vi" | "en"` from
scratch. They're structurally identical today (deliberate Track A/B split per plan: the dropdown was built by
a separate design agent with no knowledge of Track B's contract), but nothing enforces they stay in sync — if
a third locale is ever added to `SUPPORTED_LOCALES`, this file drifts silently with no compile error (unions
are structurally compatible, TS won't complain). `language-switcher.tsx:6` re-imports `Locale` from the
dropdown module rather than `lib/i18n/locale-config`, cementing the fork. Suggest: have
`language-dropdown-data.ts` import `Locale` from `@/lib/i18n/locale-config` instead of redeclaring it — Track A
already lives under `components/`, which can safely depend on `lib/i18n` without violating the "no i18n wiring
in Track A" rule (it's a type-only import, not behavior).

**3. The language switcher's own aria-labels are hardcoded Vietnamese, un-translated**
`components/language-dropdown/language-dropdown.tsx:55` (`aria-label="Chọn ngôn ngữ"`) and
`language-dropdown-list.tsx:19` (same string) never route through `next-intl`. An English-locale screen-reader
user still hears "Chọn ngôn ngữ" on the one control whose entire job is language selection. Low severity (it's
a single aria-label, not visible text) but notable because it's the intended demo of the multilingual feature.
This is a Track A/B integration gap the plan's phase-04 (integration) should have caught — Track A explicitly
declared "no i18n wiring, that's Track B" but Track B (language-switcher.tsx) only wired `onSelect`/`value`,
not these internal strings, because they're not exposed as props.

**4. Redundant defensive fallback in `site-header.tsx`**
`site-header.tsx:6,125`: `isLocale(locale) ? locale : DEFAULT_LOCALE` guards a value from `getLocale()`, which
is itself sourced from `i18n/request.ts` → `loadPreferredLocale()`, already guaranteed to return a valid
`Locale`. Harmless, but it's dead defensive code that slightly obscures the actual contract (KISS nit, not
blocking).

## What's solid

- **Path traversal**: `isLocale()` gates every value before it reaches the dynamic `import(`../messages/${locale}.json`)` in `i18n/request.ts:12` — confirmed via dedicated test (`locale-config.test.ts:22`, `'../en'` rejected). No injection surface.
- **RLS**: `20260716090000_profile_language.sql` adds a strictly owner-scoped `for update ... using (auth.uid() = id) with check (auth.uid() = id)` policy, layered correctly on top of the pre-existing select-only policy in `profile_schema.sql`. Column has a `check (language in ('vi','en'))` constraint mirroring `SUPPORTED_LOCALES` — good belt-and-suspenders against bad writes even if app-layer validation is bypassed.
- **Server/client boundary**: `setLocale` is a proper `"use server"` action; `LanguageSwitcher` client component only calls it and `router.refresh()`s — no leaking of Supabase client to the browser.
- **Error handling**: DB failure in `setLocale` is caught, logged via `console.error`, and swallowed without throwing — correct tradeoff since the cookie write (the guest-visible part) already succeeded; tested explicitly (`set-locale-action.test.ts:86`).
- **Cookie-set-during-render guard**: `load-preferred-locale.ts:32-37` wraps `cookieStore.set` in try/catch for the RSC-render-is-read-only case, with a clear comment explaining why — and it's tested (`load-preferred-locale.test.ts:111`).
- **Message parity**: `message-keys.test.ts` diffs the full key sets between `vi.json`/`en.json`; manual read confirms real parity today, and intentionally-identical values (event date, venue name, English proverb) are deliberate, not missed translations.
- **No secrets, no leaked stack traces, no string-built SQL** anywhere in scope.
- **Mock-auth branch**: `TODO(supabase)` comments consistently mark the mock bypass in both `setLocale` and `loadPreferredLocale` — easy to grep-and-remove later.
