# Calibrate Mode — Knowledge Base Tuning

Analyze historical data and propose parameter adjustments to improve estimation accuracy.

## Prerequisites

- **Recommended**: ≥2 historical entries for statistical calibration (uses CalibrationEngine)
- **Single-entry mode**: If only 1 entry exists, switch to exploratory analysis:
  - Skip CalibrationEngine (insufficient data for statistical aggregation)
  - Manually analyze: back-calculate base efforts, compare role splits, check multiplier fit
  - Label all proposals as "low confidence (1 sample)" in the report
  - Still require user approval per change
- If 0 entries: abort with message suggesting `/agentic-estimate import` first

## Workflow

### 1. Load Historical Data

```python
from agentic_estimate.utils.historical_data_loader import load_all_entries
entries = load_all_entries()
# Each entry is a dict with keys: version, type, project (name, date, domain, tech_stack, team_size, team_experience), estimate (total_md, tasks[])
```

Report counts: N accepted entries, M actual entries, date range. Access project info via `entry['project']['name']` and `entry['project']['date']`.

### 2. Run Analysis

**If ≥2 entries** (statistical mode):

```python
from agentic_estimate.utils.calibration_engine import CalibrationEngine
engine = CalibrationEngine()
report = engine.analyze(entries=entries)
```

**If 1 entry** (exploratory mode):
- Read the entry and all KB YAML files
- For role-split-heuristics: compare entry's per-category role percentages against KB defaults
- For base-efforts: if feature counts available, back-calculate per-feature base effort:
  `base = actual_md_per_feature / (complexity × tech × experience × team_size × (1+buffer))`
- For estimation-thresholds: check if QA/testing percentages align with entry data
- For management-overhead: compare PM/BrSE percentages against entry totals
- Present findings with explicit assumptions used in back-calculation

`CalibrationReport` (statistical mode) contains:
- `diffs` — list of `ParameterDiff` (proposed changes exceeding significance threshold)
  - Fields: `category`, `key`, `current`, `proposed`, `change_pct`, `samples`, `median`, `iqr`, `confidence`, `sources`
- `unchanged` — list of `str` (parameter paths within threshold)
- `warnings` — list of `str` (data quality issues)

### 2b. Gap Analysis

After running analysis (statistical or exploratory), check for KB coverage gaps.

**Read the "KB Coverage Gaps" section** from `references/historical-summary.md` (auto-generated by compile script). This section reports:

1. **Task types in historical but missing from KB base-efforts** — e.g., `data_migration`, `integration`, `management`. These are real categories the team estimates but the KB has no base effort for. Consider adding them to `base-efforts-man-days-per-task-type.yaml`.

2. **Task types in historical but missing from KB role-splits** — same types lack role distribution heuristics. Consider adding them to `role-split-heuristics.yaml` using observed historical splits.

3. **KB task types with zero historical samples** — e.g., `payments`, `authorization`. No validation data exists. Flag as low-confidence during estimation.

4. **Roles observed but undefined in KB role-splits** — e.g., `brse` appearing in `crud` tasks but KB role-split for `crud` only defines `be/fe/qa_manual`. Consider expanding role-split definitions.

**For each gap, propose to the user:**
- **Add new type**: Create base-effort and role-split entries derived from historical median values
- **Defer**: Not enough samples yet (< 2), revisit after more imports
- **Reclassify**: Historical type should map to existing KB type (e.g., `management` → `management_overhead`)

Include gap proposals in the report alongside parameter change proposals (Step 3).

**BrSE and Infra separation**: Role splits cover dev effort only (BE/FE/QA/Design). BrSE and Infra are calculated separately:
- **BrSE**: Use `management_overhead.brse.tiers` (tiered by seniority + comtor). Do NOT add brse to per-task role splits.
  - `middle_solo` (1:4): ~25% of dev MD — BrSE middle, no comtor
  - `middle_with_comtor` (1:7): ~14% of dev MD — BrSE middle + 1 comtor (default)
  - `lead_with_comtor` (1:10-12): ~9% of dev MD — BrSE lead/PSM + 1 comtor
- **Infra**: Use the `infrastructure` task type (infra: 1.0). Do NOT mix infra into dev task splits.
- **Exception**: `documentation` and `management` task types include brse as a primary role (spec creation, coordination), not as overhead.

### 3. Present Report

Display to user:

**Summary**: Total entries analyzed, date range, data quality warnings.

**Proposed Changes** (table):

| Category | Parameter | Current | Proposed | Change | Confidence | Samples |
|----------|-----------|---------|----------|--------|------------|---------|
| base_effort | crud | 3.0 | 3.5 | +17% | high | 18 |
| buffer | default | 15% | 20% | +33% | medium | 8 |

**Unchanged Parameters**: List briefly (these are within significance threshold).

**Warnings**: Show any data quality concerns (few samples, high variance, outliers).

### 4. Interactive Approval

For each proposed change, ask user:
- **Approve** — accept proposed value
- **Reject** — keep current value
- **Modify** — user provides custom value

Bulk options:
- "Approve all high-confidence changes"
- "Reject all" (review only, no changes)

### 5. Apply Approved Changes

```python
engine.apply_changes(approved_diffs)
```

This updates the relevant YAML files in `knowledge-base/`. For role splits, `apply_changes()` renormalizes sibling roles so the category sums to 1.0. Show changes via `git diff knowledge-base/` — this covers both direct changes and renormalization side effects.

### 6. Recompile Knowledge Base

```bash
python3 scripts/compile-knowledge-base.py
```

Confirm both `references/knowledge-base.md` and `references/historical-summary.md` are regenerated.

## Configuration

Tuning parameters live in `knowledge-base/historical/calibration-config.yaml`:

| Parameter | Default | Purpose |
|-----------|---------|---------|
| `weights.actual` | 3.0 | Weight for actual entries (high signal) |
| `weights.accepted` | 1.0 | Weight for accepted estimates |
| `weights.recency_decay` | 0.9 | Decay factor per year (newer = more weight) |
| `thresholds.min_samples` | 2 | Minimum entries to propose a change |
| `thresholds.significance_pct` | 10 | Minimum % difference to flag as a change |
| `algorithm` | weighted_median | Aggregation method |
| `blending.prior_strength` | 5 | Bayesian prior pseudo-samples |

## Engine Limitations

CalibrationEngine performs **statistical aggregation** (weighted median across entries). It does NOT:
- Back-calculate per-feature base efforts from category totals
- Analyze user-provided domain knowledge (e.g., "QA is 30% in our company")
- Propose new KB sections (e.g., feature_composite)

For these, use **exploratory mode** (manual analysis with user approval).

## Rollback

KB YAML files are version-controlled. To revert calibration changes:

```bash
git diff knowledge-base/       # review what changed
git checkout -- knowledge-base/ # revert all KB changes
python3 scripts/compile-knowledge-base.py  # regenerate references
```
