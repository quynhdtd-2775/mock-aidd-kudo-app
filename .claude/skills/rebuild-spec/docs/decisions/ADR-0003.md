# ADR-0003 — Component per-lang placement: single-source translate model

**Status:** Accepted (v23.0.0)
**Supersedes:** ADR-0002 (source-vs-derived-view model)
**Date:** 2026-06-30

---

## Context

ADR-0002 introduced a source-vs-derived-view split to solve the per-lang placement problem:
per-component runs always wrote `docs/components/<name>/` (lang-agnostic source), while the
aggregate rebuilt `docs/<primary>/components/<name>/` as a *derived view* each run.

Two problems emerged from real non-en single-lang usage:

1. **Duplicate tree (the core failure).** A non-en single-lang repo (e.g. `primary_lang: vi`,
   no translations) resolves its docs root to `docs/vi/` (per `_lang_lib.resolve_docs_root`).
   With ADR-0002 the component source lived at `docs/components/<name>/` (root), while the
   derived view was at `docs/vi/components/<name>/`. This produced TWO parallel component trees
   that diverged on every per-component run — the source grew but the view was only refreshed by
   `--aggregate`, causing stale state and reader confusion.

2. **Rung-3 "dùng tạm" fallback.** ADR-0002's rung-3 projected the primary-lang base as a
   temporary stand-in whenever a component had no `<L>` mirror, emitting a non-blocking `[WARN]
   lang_fallback`. This masked missing translations silently; there was no automatic path to
   produce a real translated copy. Rung-3 is deleted in v23.

3. **Derived-view code surface.** `_component_view_lib.py` (atomic temp-dir swap) and the
   projection loop in `_component_placement_lib.py` were non-trivial stateful machinery
   maintained separately from the existing translation pipeline, which already handled the
   per-lang slice problem correctly for core/system artifacts.

---

## Decision

### Single-source placement via `resolve_docs_root`

Component docs are written **once** to the language-resolved source root. There is no derived
view.

| Lang | Source path | Who writes |
|------|-------------|-----------|
| `en` single-lang | `docs/components/<name>/` | per-component `--root`/`--batch` runs |
| non-en (primary or per-lang) | `docs/<primary>/components/<name>/` | per-component `--root`/`--batch` runs |

`_path_lib.resolve_component_paths(project_root, plan_dir, root_arg, primary_lang)` passes
`primary_lang` to `_lang_lib.resolve_docs_root(lang, primary_lang, multilang)` — the SAME
resolver used by core and system artifacts. This is the single source of truth for the
component docs path.

### Secondary languages — translate-per-component (Path B reuse)

Secondary language content at `docs/<L>/components/<name>/` is produced by the **existing**
translation pipeline (`_translation_sync_lib` Path B), scoped to the component's docs subtree.
`_DOC_AREAS` includes `"components"` so the auto-sync gate picks up changed component artifacts
exactly as it does for core/system artifacts.

There is **no rung-1/rung-2/rung-3 projection**. Every secondary-lang component doc is a real
translation, not a copy of primary content.

### Derived-view code deleted

`_component_view_lib.py` and the projection entry-point in `_component_placement_lib.py` are
deleted. `metadata.json → deletions` records both. The test file
`tests/test_component_placement.py` is also removed.

---

## Rejected alternatives

### (a) Keep ADR-0002 source-vs-derived-view

**Rejected because:** duplicate tree problem on non-en single-lang repos is structural — the
ADR-0002 model has no way to unify the root source and the lang-namespaced view without
reintroducing the ping-pong risk it was designed to avoid.

### (b) Lazy-only sync (translate on first demand)

Translate component artifacts only when `--aggregate` is called, rather than at per-component
promote time.

**Rejected because:** this defers translation to a later pipeline stage that already has a
different job (system synthesis). It also breaks the `auto_sync` contract that other artifact
layers follow, requiring a special code path for components.

### (c) Keep rung-3 "dùng tạm" fallback

**Rejected because:** rung-3 silently masked missing translations. A developer reading
`docs/vi/components/<name>/` would see primary-lang content with no indication it was
untranslated. Deleting rung-3 forces explicit translate runs and surfaces gaps clearly.

---

## Consequences

- **en single-lang repos:** `docs/components/<name>/` path is byte-identical to v22 and earlier.
  No migration needed.
- **non-en repos (v20/v22 migration):** existing `docs/components/` source is migrated to
  `docs/<primary>/components/` by `_component_migrate_lib.migrate_components_to_lang`, which runs
  automatically on the first `--aggregate` call for repos where `primary_lang != en` and the old
  root layout still exists. Guarded by sentinel `docs/<primary>/.components-migrated-v23`. See
  `references/multi-component-runbook.md` § "v23 one-time component-tree migration (P07)".
- **Differ branch:** if both `docs/components/` and `docs/<primary>/components/` exist and files
  differ (manual edits), the migration emits `[WARN]` and does NOT move — both trees are kept
  until the user resolves the conflict manually. This is a `DONE_WITH_CONCERNS` posture: the
  sentinel is NOT written, so the migration re-attempts on next `--aggregate`.
- **Translation cost:** secondary-lang component docs are now real translations (Path B cost),
  not free copies of the primary. This is intentional — `--lang <L> --root <name>` produces
  the translation explicitly; auto-sync only re-translates CHANGED artifacts, not the whole tree.
- **Reading-guide (nav):** `build_navigation.py` generates READMEs at the new per-lang component
  path. No `docs/components/` README is written for non-en repos.
- **`SYNTHESIS_FORMAT_VERSION`** bumped from `21.0.0` to `22.0.0` to trip the `[WARN]
  stale_digest` guard on existing aggregate state files, forcing re-synthesis after migration.
