# ADR-0002 — Component per-lang placement: source-vs-derived-view model

> **Superseded by [ADR-0003](./ADR-0003.md) (v23.0.0).** The derived-view model is removed;
> the single-source translate model is now in effect. This document is kept as immutable history.

**Status:** Superseded (v20.0.0 → superseded v23.0.0)
**Supersedes:** ADR-0001 (deferred flatten of bilingual component layout)
**Date:** 2026-06-26

---

## Context

In a per-lang rebuild-spec repo, components must be presented at `docs/<L>/components/`
with the appropriate language slice. Two constraints make this non-trivial:

1. **Per-component runs are lang-agnostic.** `_path_lib.resolve_component_paths` always
   writes `docs/components/<name>/` — this is documented and caller-visible. Making it
   lang-aware would break `--root` back-compat and spread language logic into the hot path.

2. **Ping-pong risk.** Any design that moves component content into `docs/<L>/components/`
   will have it recreated at `docs/components/<name>/` on the next `--root` run —
   producing drift between the two locations.

3. **Data-loss risk (the v15 lesson).** A bilingual component has its `<L>/ ` mirror as a
   CHILD of its own dir. Any `rmtree(dest)` or move that uses the component dir as the
   destination will destroy the live mirror. This was the root cause of the v15 data-loss
   regression.

---

## Decision (D1): Source-vs-derived-view

| Path | Role | Owner | Lang-aware? |
|------|------|-------|------------|
| `docs/components/<name>/` | **Source of truth** | per-component runs (`--root`/`--batch`) | No (unchanged) |
| `docs/<L>/components/<name>/` | **Derived view** (rung-selected slice) | aggregate (`--aggregate`) only, rebuilt atomically each run | Yes (per-component rung) |

These two paths have distinct ownership and never conflict. Per-component runs write only to
the source; the aggregate writes only to the derived view, reading the source as input.

### Rung selection (per component `C`, target language `L`)

| Rung | Condition | What is projected |
|------|-----------|-------------------|
| 1 | `docs/components/<C>/<L>/` mirror dir exists | Flatten `<C>/<L>/*` into view root (mirror wins on collision); exclude other-lang subdirs |
| 2 | component `primary_lang == L` | Copy base, excluding sibling lang-mirror subdirs |
| 3 | neither | Copy primary base as TEMP stand-in + emit `[WARN] lang_fallback` (non-blocking) |

### Atomic full rebuild

The entire `docs/<L>/components/` view is rebuilt via a temp dir + atomic rename each run:

```
build → docs/.<L>-components.tmp/
rename old → docs/.<L>-components.old/
rename tmp → docs/<L>/components/
rmtree .old
```

A crash leaves either old or new view intact. Stale orphan dirs (component removed from
source) are pruned for free because the view is always built fresh from the current source.

---

## Rejected alternatives

### (a) Per-component runs become lang-aware

Make `resolve_component_paths` write `docs/<L>/components/<name>/` directly.

**Rejected because:**
- Breaks documented `--root` byte-compat (`_path_lib` docstring).
- Spreads language resolution into the per-component hot path.
- Still needs a fallback story when the component has no `<L>` content yet.
- Violates KISS + the `--root` single-source contract.

### (c) Sentinel-gated MOVE (the v15 design)

Move each rung-selected slice into `docs/<L>/components/<name>/`.

**Rejected because:**
- This is exactly the design that caused data loss: the `<L>/` mirror is a CHILD of the
  component dir, so `rmtree(dest)` or `os.rename(src, dest)` where dest already exists
  destroys the live source.
- Creates ping-pong: per-component `--root` runs recreate `docs/components/<name>/` →
  drift between source and moved copy.
- Strands the other language (a bilingual component's non-`L` primary is discarded).

### (b) Keep current "move whole container + classify-only" (v15 model)

Relocate the entire `docs/components/` container to `docs/<L>/components/` via
`LANGUAGE_LAYERS` membership.

**Rejected because:**
- Never surfaces the `<L>` slice: a bilingual component's `<L>` content stays buried at
  `docs/<L>/components/<C>/<L>/...` (one tier too deep).
- Creates ping-pong: the next `--root` run recreates `docs/components/<name>/`.
- Container-level move requires per-component content to already be at the target —
  incompatible with the "project only what's needed" goal.

---

## Consequences

- `MOVED_LAYERS` replaces `LANGUAGE_LAYERS` for flip/relocate operations (`components`
  excluded — it never moves via the layout flip).
- `LANGUAGE_LAYERS` is kept as a backward-compat alias (still includes `components`) for
  callers like rollback that need to handle v15-era partial states.
- Legacy trees with `docs/<L>/components/` from a prior v15 run are converged back to
  `docs/components/` (source) on the first aggregate run via `_converge_components_to_source`.
- The derived view is a real committed copy (Q1 decision 2026-06-26): readable on GitHub;
  accepted repo-bloat + git churn per aggregate run.
- ADR-0001's "deferred flatten" stance is superseded: the flatten IS done (rung-1 mirror
  content is flattened into the view root), but only in the derived view, never in the source.
